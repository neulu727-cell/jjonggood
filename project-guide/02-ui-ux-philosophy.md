# UI/UX 철학 — 오너의 피드백에서 추출한 원칙

## 핵심 철학: "빠르고 간결하게"

이 앱은 미용사가 강아지 미용하면서 한 손으로 조작하는 도구다.
불필요한 입력은 제거하고, 한 화면에 모든 정보를 보여준다.

---

## 원칙 1: 필드 최소화

### 규칙
- 등록 폼은 **4개 이하** 필드로 유지
- "나중에 필요할 수도 있는" 필드는 넣지 않는다
- 수정 폼에서만 추가 필드 노출

### 적용 사례
| 폼 | Before | After |
|----|--------|-------|
| 고객 등록 | 전화번호, 이름, 견종, 몸무게, 나이, 특이사항, 메모 (7개) | 전화번호, 이름+견종, 몸무게, 메모 (4개) |
| 예약 등록 | 서비스, 소요시간, 상담금액, 결제금액, 결제방법, 요청사항, 메모 | 서비스, 소요시간, 금액, 메모 |

### "이름+견종" 합체 입력
신규 등록 시 한 필드에 `루미 말티푸` 입력 → 자동 파싱:
1. CONFIG.breeds 리스트에서 긴 것부터 매칭
2. 매칭 안 되면 마지막 공백 기준 분리
3. 공백 없으면 전체가 이름, 견종 빈칸

수정 폼에서는 이름/견종 별도 필드로 표시.

## 원칙 2: 빈틈 없는 정보 밀도

### 규칙
- **스크롤 없이** 모든 핵심 정보를 볼 수 있어야 한다
- 빈 공간은 "짜치는" 느낌을 준다 → 압축
- PC에서는 반드시 2열 레이아웃 사용
- 관련 정보는 한 줄에 합치기 (예: "몸무게"를 "반려동물" 행에 합침)
- **박스/카드로 위계를 잡지 말고, 타이포그래피(굵기·크기·색상)로 위계를 잡는다**
- 불필요한 label:value 테이블 형식 금지 — 인라인 문장으로 표현

### 핵심: "박스가 아니라 텍스트가 일한다"
```
❌ 나쁜 예 — 박스로 위계를 잡음 (공간 낭비)
┌──────────┐ ┌──────────┐ ┌──────────┐
│    12    │ │  850,000 │ │  70,833  │
│   방문   │ │   매출   │ │   평균   │
└──────────┘ └──────────┘ └──────────┘
일시     2026-03-13 10:00~12:00     ← label:value 테이블
서비스   전체얼컷 120분 / 9mm
금액     60,000원
상태     [완료]

✅ 좋은 예 — 타이포그래피로 위계 (같은 정보, 1/3 공간)
**12**회 방문 · 매출 **850,000**원
첫 01.15 · 최근 03.13(0일전) · 주기 28일

▌현재 예약
  03.13(목) 10:00~12:00 · [완료]
  전체얼컷 120분/9mm · 60,000원
  [수정] [되돌리기]
```

### 원칙
| 패턴 | 사용 금지 | 사용 권장 |
|------|----------|----------|
| 통계 | 큰 숫자 박스 3개 | 인라인 "N회 · N원" |
| 상세 정보 | label:value 테이블 4~5행 | 밀도 높은 2~3줄 문장 |
| 구분 | 테두리 박스, 배경 카드 | 좌측 액센트바, 들여쓰기 |
| 액션 | full-width 버튼 | 텍스트 링크 또는 인라인 소형 버튼 |
| 불필요한 지표 | "평균 단가" 같은 파생 값 | 제거 — 핵심만 남긴다 |

## 원칙 3: 워크플로우 순서 지키기

### 규칙
- 입력 시점이 아닌 **결정 시점**에 정보를 받는다
- 예약 등록 시 결제방법을 묻지 않는다 (아직 미용 안 했으니까)
- "미용 완료" 클릭 시 결제방법 팝업이 뜬다

### 결제 흐름
```
예약 등록 → 금액만 입력
미용 완료 버튼 클릭 → 결제방법 선택 팝업 (카드/현금)
                    → 선택 즉시 상태 변경 + 결제방법 저장
```

## 원칙 4: 메모는 "추가" 방식

### 규칙
- 메모 수정 = 기존 내용 **덮어쓰기** 아님
- "메모 추가" 버튼으로 기존 메모 뒤에 줄바꿈 후 추가
- 한번 쓴 메모는 실수로 날아가면 안 된다

### 로직
```javascript
// 기존 메모 가져오기 → 새 텍스트 앞에 붙이기 → 저장
const existing = await fetch(`/api/customer/${id}`).then(r => r.json());
const merged = existing.memo
    ? existing.memo + '\n' + newText
    : newText;
await fetch(`/api/customer/${id}`, {
    method: 'PUT',
    body: JSON.stringify({ memo: merged })
});
```

### "메모 추가" vs "메모 저장"
- "저장" → 덮어쓰기 느낌 (위험)
- "추가" → 기존 것에 더하는 느낌 (안전) ✅

## 원칙 5: 통합 상세 시트 (Unified Detail)

### 규칙
- 예약 상세와 고객 상세는 **하나의 통합 시트**(unifiedDetailSheet)로 처리
- 어디서 진입하든(타임라인, 고객목록) 같은 통합 페이지가 열린다
- 타임라인에서 진입 → 해당 예약이 "현재 예약"으로 하이라이트
- 고객목록에서 진입 → 하이라이트 없이 전체 정보 표시
- 예약 이력은 별도 시트가 아닌 **아코디언**으로 그 자리에서 펼침

### z-index 체계
| 레이어 | z-index | 용도 |
|--------|---------|------|
| 바텀시트 기본 | 100 | 고객 등록, 예약 등록 |
| 통합 상세 | 110 | 예약+고객 통합 상세 |
| 전화 팝업 | 200 | ADB 수신 알림 |
| 토스트 | 300 | 알림 메시지 |

### 시트 열기 흐름
```
타임라인 예약 클릭 → showReservationDetail(rid, customerId)
                   → customer API 1회 호출
                   → renderUnifiedDetail(c, activeRes)

고객목록 클릭 → showCustomerDetail(cid)
              → customer API 1회 호출
              → renderUnifiedDetail(c, null)
```

**핵심**: API는 항상 1회만 호출. 예약 데이터는 customer API 응답의 reservations 배열에서 탐색.

## 원칙 6: 강아지 중심 (Pet-First)

### 규칙
- 고객 = 보호자 (phone으로 식별)
- 핵심 단위는 **강아지** (pet_name + phone이 UNIQUE)
- 같은 보호자에 여러 강아지 → 펫 스위처로 전환
- 메모도 강아지별로 관리

### 멀티펫 구조
```
보호자 (010-1234-5678)
├── 두리 (비숑, id=26) → 개별 메모
└── 몽실 (시츄, id=30) → 개별 메모
```

### 고객 상세에서
- 펫 스위처 (pill 버튼)로 강아지 전환
- 각 강아지의 메모, 예약 이력이 별도 표시
- siblings 쿼리로 같은 전화번호의 다른 강아지 조회

## 원칙 7: 타임라인은 모든 예약을 보여준다

### 규칙
- 과거 날짜든 미래 날짜든, 모든 상태의 예약을 표시
- 캘린더 뱃지와 타임라인의 데이터가 **일치**해야 한다
- 과거 날짜에서는 빈 슬롯만 숨김

### 필터 금지
```javascript
// ❌ 과거 날짜에서 완료만 보여주기 (캘린더와 불일치)
if (isPast && r.status !== 'completed') return;

// ✅ 모든 예약 표시, 빈 슬롯만 숨김
const isPast = selectedDate < today;
if (isPast) { /* 빈 슬롯만 숨김 */ }
```

## 원칙 8: 드롭다운 대신 버튼 그리드

### 규칙
- select 드롭다운은 터치 환경에서 불편
- 자주 쓰는 옵션은 **버튼 그리드**로 한눈에 보이게
- 직접 입력이 필요하면 텍스트 필드와 병행

### 적용
- 서비스 종류: 목욕, 위생, 전체미용, 부분미용, 스포팅, 가위컷, 기타
- 소요시간: 30분, 1시간, 1시간반, 2시간, 2시간반, 3시간
- 금액: 미리 정의된 금액 버튼 + 직접 입력
- 결제방법(완료 시): 카드, 현금 (2개 버튼 팝업)
